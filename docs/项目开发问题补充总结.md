# 项目开发问题补充总结

本文根据 git 历史补充总结当前项目开发过程中遇到、但还没有被现有 `docs/` 单独记录的问题。

为避免重复，以下主题不在本文展开：

- `Qwen3.5 remove-padding position_ids` 问题
- 训练期 `DAPO + vLLM` LoRA 动态注入问题
- Reward 设计本身
- LiveCodeBench 评测链路本身
- 数据构造流程本身

上面这些内容已经分别在现有文档中记录。

---

## 1. 补充结论

从 git 历史看，当前项目除算法与训练逻辑问题外，还长期受三类工程困难影响：

1. `Qwen3.5 + vLLM + LoRA` 的评测与侧载兼容性不稳定
2. 显存预算和批量参数非常敏感，配置经常需要回退和重试
3. 代码执行器与 sandbox 环境很难一次性做成稳定方案

这三类问题的共同特点是：

- 它们不总是表现为单个 Python 报错
- 更多时候是“能跑一部分，但链路不稳定”
- 最终逼着项目通过多轮脚本、配置和实现回退来逐步收敛

---

## 2. 问题一：`Qwen3.5 + vLLM + LoRA` 的评测兼容性并不成熟

这一类问题主要发生在 2026-04 初的一批提交中。

关键提交包括：

- `ef18f1a`：测试 `vLLM` 加载 `Qwen3.5 LoRA`，明确记录“发现问题”
- `053e617`：添加 `vLLM` 对 `Qwen3.5` 的支持
- `42ab3b0`：链路跑通，但出现 OOM
- `b05e3e7`：修复 `vLLM engine` 默认参数没有传入的问题
- `0c1bccd`：补上 LoRA 评测脚本
- `adb7fbd`：让 `LiveCodeBench` 支持 LoRA 侧载

从这组提交能看出，当时的困难不是一个孤立 bug，而是一串互相耦合的问题：

- `Qwen3.5` 本身属于较新的模型结构
- `vLLM` 对它的支持还在快速演进
- LoRA 又额外引入了 adapter target、评测目录、推理参数等兼容要求

项目采取的策略不是一次性“修根因”，而是分层补齐：

- 先加测试确认问题存在
- 再补模型支持
- 再修参数透传
- 再把 LoRA 评测和侧载链路接上

这说明一个很现实的工程经验：

- 在新模型上接 `vLLM + LoRA + benchmark`
- 真正困难的常常不是单个 API
- 而是推理引擎、评测脚本、adapter 目录、参数默认值一起组成的系统兼容性

---

## 3. 问题二：显存和批量配置长期处于“边跑边调”的状态

仓库里有不少提交直接反映了这一点，例如：

- `076cc4e`：限制 `vLLM` 最大显存
- `42ab3b0`：明确记录“跑通，但 OOM 了”
- `00c4588`：记录会 OOM 的参数
- `127c19d`、`41c9e2e`、`374542e`、`8072f82`：多轮参数调整
- `506b195`：使用官方推荐参数，并回退一部分对 `verl` 的改动

这类历史说明，项目里显存问题不是“偶发超一点”，而是系统性约束：

- `Qwen3.5`
- `vLLM`
- LoRA
- RL rollout
- DAPO 的 group / batch 配置

叠在一起之后，显存和吞吐的可行区间就会变得很窄。

从历史选择看，项目实际采用过几种手段：

- 给 `vLLM` 显式加更保守的显存上限
- 调整训练和生成 batch 配置
- 记录“能跑的参数”和“会 OOM 的参数”
- 在必要时回退部分框架改动，优先恢复可运行状态

这里暴露出的核心困难不是“不会调参”，而是：

- 某些参数组合即使逻辑正确，也未必有工程可行性
- 所以开发过程里必须把“是否稳定可跑”作为一等约束

这也是为什么仓库里后续 README 会专门解释 `gen_batch_size`、`train_batch_size` 以及相关指标含义。

## 4. 问题三：代码执行器和 sandbox 的工程现实比预期更复杂

这类问题主要集中在 2026-03 末的早期阶段。

关键提交包括：

- `293b820`：尝试初步的 sandbox
- `4f1b283`：尝试高并发 sandbox
- `009fa70`：明确记录“尝试沙盒未果，暂时放弃”
- `8053be8`：给数据清洗打标加并行
- `17790b7`：在执行前裁剪 testcase，并提升内存限制
- `039d7bc`：给 `code_executor` 增加 `SystemExit` 解析
- `f91f5ce`：删除 `multiprocessing`，回到单线程执行

从这条演化线可以看到几个很典型的问题：

### 4.1 sandbox 方案理论上吸引人，但落地依赖很多环境条件

项目早期显然尝试过把代码执行放进更强隔离的环境里，但最后没有继续推进。

这通常意味着至少存在下面几类阻力中的一种或多种：

- 环境权限不够
- 容器/沙盒依赖难以稳定部署
- 吞吐和复杂度不成比例

对于这个仓库的实际目标来说，最后显然选择了更务实的路线：

- 先保证执行链路可用
- 再逐步补执行保护和错误处理

### 4.2 多进程并不一定比单线程更稳

`f91f5ce` 很有代表性。

这次提交不是“进一步优化并行”，而是明确把 `multiprocessing` 删掉，回到单线程执行，理由就是某些环境下多进程会出问题。

这说明项目实际踩到的是典型的执行器困境：

- 并行更快
- 但在真实训练机、数据机、评测机环境中，稳定性可能反而更差

### 4.3 testcase 和错误类型必须做工程化裁剪

从 `17790b7` 和 `039d7bc` 可以看出，执行器后面还补了两类很重要的现实处理：

- testcase 太大时，先裁剪
- `SystemExit` 这类特殊终止，要单独识别

这说明代码任务里的 reward/标注基础设施不是“运行一下 Python”这么简单，而是需要长期处理：

- 内存
- 超时
- 退出方式
- 错误归因

## 5. 问题四：很多困难最后都是靠“先记录，再收敛”解决的

如果把整个 git 历史顺着看，会发现一个很鲜明的开发风格：

- 先测试或尝试
- 明确记录失败现象
- 再加日志、加脚本、调参数、做回退
- 最后把稳定方案写入文档

这在提交信息里表现得非常明显，例如：

- “发现问题”
- “尝试修复”
- “跑通，但 OOM 了”
- “回退”
- “最新能跑的”

这说明当前项目的很多关键能力，并不是靠一次设计直接完成的，而是靠下面这套方法逐步拿到的：

1. 尽快让问题可复现
2. 用最小修改缩小范围
3. 必要时接受临时回退
4. 一旦稳定，就补文档和说明

这套方法虽然不“漂亮”，但对当前这种跨训练、推理、评测、执行器多子系统耦合的项目来说，其实是非常现实且有效的。

---

## 6. 最终总结

除去已经有专门文档记录的主线问题，这个项目在开发过程中还反复遭遇了三类典型困难：

- 新模型与 `vLLM/LoRA` 的评测兼容性不稳定
- 显存和 batch 配置非常敏感
- 代码执行与 sandbox 难以同时兼顾隔离性、吞吐和稳定性

这些问题的价值在于，它们提醒我们：

- 当前仓库的难点不只在算法
- 很多成本其实发生在“把整条实验链路做成可稳定运行的系统”上

后续如果继续扩展模型、评测方式或执行环境，优先值得保留的经验是：

- 先做可复现与可观测
- 再做性能和复杂功能
- 对新模型保持“先验证兼容性，再扩大使用范围”的节奏
