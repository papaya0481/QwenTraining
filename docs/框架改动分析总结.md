# 框架改动分析总结

本文总结当前项目相较于原版 `ms-swift` 与 `verl` 所做的主要框架改动，重点回答两件事：

- 我们具体做了哪些工程工作
- 这些改动分别解决了什么问题

为方便阅读，本文只保留基线 commit 的短哈希：

- `ms-swift` 基线：`b59a105e`
- `verl` 基线：`516657fa`

---

## 1. 总体结论

相较于原版框架，我们并不是只做了少量配置调整，而是围绕训练闭环补了几类关键能力：

- 打通了 `LoRA` 训练、保存、加载、评测之间的链路
- 补齐了 `VeRPO` 所需的 reward 与 advantage 计算能力
- 增强了评测链路，尤其是 `ms-swift` 上的 `vLLM` 与 `LiveCodeBench` 支持
- 提升了 checkpoint、dtype、dataloader state 等训练基础设施的稳定性
- 补了一部分兼容性与可观测性能力，降低调试和迁移成本

如果从工程性质上看，这些工作更接近“把上游通用框架改造成适合我们当前代码强化学习任务的可用版本”，而不是简单地在原版上打几个补丁。

---

## 2. `ms-swift` 侧主要做了什么

`ms-swift` 这部分改动主要集中在评测链路，尤其是 LoRA 模型评测和 `vLLM` 推理支持。

### 2.1 补上了 LoRA 评测目录与结果管理

原版评测结果目录更偏向基础模型场景。我们扩展后，评测结果目录可以自动带上 adapter 信息，避免不同 LoRA checkpoint 或不同 adapter 之间互相覆盖结果。

这项工作的价值是：

- 让 LoRA 评测结果具备可区分性
- 降低多轮实验后结果混淆的概率
- 让训练 checkpoint 与评测产物之间更容易建立对应关系

### 2.2 增强了 `vLLM` 初始化与推理兼容性

我们给 `vLLM` 推理链路补了几层兼容：

- 当 `patch_auto_config` 导致配置类型不兼容时，自动回退重试
- 只在采样参数非空时才向 `vLLM` 传参，避免无意义或错误的参数注入
- 在评测路径中支持 `infer_backend=vllm`

这项工作的本质是把原本更偏理想路径的接入，改成了更接近真实实验环境的鲁棒实现。特别是在 `Qwen3.5` 一类模型或多模态相关配置下，这类保护逻辑能明显减少“框架能跑但评测初始化失败”的问题。

### 2.3 为评测链路增加了 OOM 退避与 fallback
> [!WARNING]
> 该功能实际未开启。

我们在评测时没有假设显存总是足够，而是加入了更保守的资源策略：

- 自动估算更安全的 `gpu_memory_utilization`
- `vLLM` 初始化失败时逐步降低并发和上下文长度
- 多次尝试仍失败时回退到 `TransformersEngine`

这说明我们做的不只是“支持 vLLM”，而是把它做成了一个在实际训练机上更容易落地的方案。这样可以减少因为显存波动、checkpoint 大小变化、评测批量设置变化而导致的整轮评测失败。

### 2.4 增强了 `LiveCodeBench` 评测结果导出

我们额外导出了逐样本 CSV，包含：

- 样本索引
- 是否正确
- 错误原因
- 提取出的代码块

这项工作非常偏向实验分析效率。原始 benchmark 汇总分数只能告诉我们“整体好不好”，但逐样本 CSV 能直接支持：

- 错误类型分析
- bad case 回看
- 不同 checkpoint 的样本级对比

对于代码任务，这种粒度的评测产物比单纯看 aggregate score 更有用。

### 2.5 把训练态参数更完整地透传到了评测态

我们在 trainer 与 EvalScope 的对接里，补充了多类模型侧参数透传，例如：

- `infer_backend`
- `checkpoint_root`
- `global_step`
- LoRA adapter 相关信息
- 一组 `vLLM` 参数

这项改动解决的是“训练配置和评测配置脱节”的问题。训练过程中产生的 checkpoint、adapter、step 信息，能够更自然地进入评测链路，从而形成更完整的实验闭环。

## 3. `verl` 侧主要做了什么

`verl` 这部分改动范围更大，核心集中在四条线：

- `VeRPO` 算法接入
- `LoRA` 保存加载链路增强
- checkpoint / dtype / dataloader state 基础设施完善
- reward manager 与 rollout 可观测性增强

### 3.1 把 `VeRPO` 真正接进了训练框架

这部分不是简单新增一个 reward 脚本，而是把 `VeRPO` 需要的关键环节都接进了 `verl`：

- 新增 `VeRPO` reward manager
- 在 reward loop 中支持优先走 `run_batch(...)`
- 新增 `VERPO` advantage estimator
- 增加 `verpo_turn_beta`、`verpo_turn_norm_by_std` 等算法配置
- 在 trainer 侧把 `dense_reward`、`traj_reward` 等字段传给 advantage 计算

这说明我们做的是“算法接入”而不是“局部实验脚本”。  
`VeRPO` 的关键不在于单样本打分，而在于组内统计量和双层 advantage 结构。原版框架并没有完整承载这套逻辑，我们补的正是这部分框架能力。

### 3.2 让 reward loop 支持组级别打分

我们显式支持了 reward manager 的批量执行入口 `run_batch(...)`，使同一个 prompt 下的多条 response 可以一起计算 reward。

这项改动的重要性在于：

- 保住组内统计量，例如 testcase 级通过率 `rho_j`
- 让 `dense_reward` 变成真正的 group-aware 信号
- 避免退化成逐样本打分后，`VeRPO` 只剩下表面形式

从算法正确性角度看，这是本轮改动里很关键的一项。

### 3.3 重做了 LoRA 的保存与加载兼容

这是 `verl` 中另一块很重要的工作。我们围绕 LoRA 做了几类互相配合的修改：

- 保存时额外导出标准 PEFT 格式的 `lora_adapter`
- 加载时支持从不同目录布局自动解析真实 adapter 路径
- 对 `vLLM` 读取 LoRA tensor 时做 key 规范化
- 为 FSDP2 场景调整 LoRA 参数收集方式

这一组改动解决的是“训练能存、评测能读、不同后端都能认”的问题。

如果没有这些工作，就很容易出现下面这些断点：

- 训练保存的 LoRA 目录结构和评测/推理要求不一致
- `FSDP1/FSDP2` 下保存行为不同
- `PEFT`、`vLLM`、训练侧对 LoRA key 命名的理解不一致

换句话说，我们补的是一条端到端的 LoRA 工程链路，而不是单点修复。

### 3.4 让 checkpoint 更适合当前训练场景

我们把 dataloader state 的保存和恢复改成了可选项，而不是始终强制保存 `data.pt`。

这项修改的意义主要有三点：

- 减少不必要的 checkpoint 产物
- 降低某些环境下 resume 的脆弱性
- 让训练恢复逻辑更适配我们实际使用方式

对大规模实验来说，这类基础设施细节虽然不显眼，但会直接影响训练稳定性和排障成本。

### 3.5 补了 reward manager 配置解析能力

我们把 reward manager 的参数装载逻辑做得更灵活，兼容了不同风格的 `reward_kwargs` 写法，并且只把 manager 实际支持的参数传进去。

这项工作的价值是：

- 降低自定义 manager 接入成本
- 减少配置层和实现层之间的耦合
- 避免因为参数风格变化而引入运行时错误

对于经常需要试验 reward 设计的项目来说，这是一项很实用的工程改造。

### 3.6 增强了训练链路的可观测性与小型兼容修复

除主线改动外，我们还做了一些“小但有价值”的补强：

- 给单轮 agent loop 增加 trace 装饰器，便于观察 rollout 行为
- 增加 adapter 路径解析的测试用例
- 保存 LoRA 失败时输出更完整的异常信息

这些改动不一定直接改变算法，但会明显影响问题定位效率和后续维护成本。

---

## 4. 这些改动合起来说明了什么

如果把 `ms-swift` 和 `verl` 两边放在一起看，我们的工作可以概括为三层。

### 4.1 第一层：把训练、保存、评测打通了

这层工作主要体现在 LoRA 链路、checkpoint 链路和评测链路的衔接上。  
原版框架各自具备一部分能力，但并不完全适合我们当前“代码任务 + LoRA + RL + benchmark 分析”的组合场景。我们做的，是把这些能力拼成一个完整闭环。

### 4.2 第二层：把上游框架补成了适合 `VeRPO` 的版本

`VeRPO` 不是只加一个 reward 函数就能跑起来，它要求：

- 组级别 reward 统计
- 额外 reward 信号字段
- 对应的 advantage 计算方式
- 训练配置层面的参数接入

我们已经把这些部分补到了 `verl` 内部，因此现在的仓库不只是“借用原版 PPO/DAPO”，而是已经具备了承载 `VeRPO` 训练的框架能力。

### 4.3 第三层：把实验系统做得更稳、更能分析

这层改动更多体现在：

- `vLLM` 的鲁棒初始化
- OOM 退避与 fallback
- 逐样本 CSV 导出
- trace 和测试补充

这些工作未必最显眼，但它们会显著减少实验中断、降低排障成本，并提高后续分析效率。

---

## 5. 最终总结

相较于原版 `ms-swift` 和 `verl`，我们当前仓库完成的不是单点 patch，而是一组面向代码强化学习任务的系统性改造。

从结果上看，这些工作主要落在以下几个方向：

- 在 `verl` 中补齐了 `VeRPO` 所需的 reward 与 advantage 框架能力
- 在 `verl` 和 `ms-swift` 之间打通了 `LoRA` 的训练、保存、加载、评测链路
- 增强了 `ms-swift` 的 `vLLM` 评测能力和 `LiveCodeBench` 结果分析能力
- 完善了 checkpoint、dataloader state、dtype、异常处理等训练基础设施
- 补充了测试和可观测性能力，使后续维护与实验分析更可控

因此，更准确的描述不是“我们在原版框架上改了一些细节”，而是：

我们围绕当前代码任务 RL 训练场景，对上游框架做了较完整的任务化适配和工程化增强。
